Introduction

Questo è lo Sprint2 del progetto BoundaryWalk con HTTP

Lo Sprint1 ha prodotto:
  • un programma che realizza il core applicativo unibo.http.Appl1HTTPSprint1
  • un test automatizzato (non soddisfacente) per il core applicativo unibo.http.Appl1HTTPSprint1
    • Il sistema costruito si riduce a un main program che usa un supporto per le comunicazioni:

    Requirements (TESTO ORIGINALE)

    Scrivere un programma Java che possa inviare al VirtualRobot23 (attraverso una Console remota):
  • un comando start per attivare il VirtualRobot in modo che percorra il bordo perimetrale della stanza rappresentata ne La scena di WEnv
  • un comando stop per bloccare l’azione del VirtualRobot.
  • un comando di resume per riattivare l’azione del VirtualRobot.
    • Goal dello Sprint2: comandi start/stop/resume e console

    Requirement analysis

  • La logica applicativa deve poter operare in seguito a comandi inviati da CmdConsole remota.
  • La logica applicativa non deve partire in modo autonomo, ma deve essere attivata dal comando start.
  • KEY POINT-Req1 Il comando di stop deve poter essere gestito dalla la logica applicativa (attualmente proattiva) mentre sta muovendo il robot
  • KEY POINT-Req2 Non sono previste più console
  • KEY POINT-Req3 Non ci sono requisiti sulla sicurezza

    • Problem analysis

    • La console remota non fa parte della logica applicativa, ma ne provoca l'esecuzione di azioni quali star/stop/resume.
      KEY POINT-AP1 Viene naturale modellare la logica applicativa come un POJO che realizza l'interfaccia 
                   interface IAppl1Core {
                public void start() throws Exception;
                public void stop();
                public void resume();
             }
    • Il metodo start innesca l'azione proattiva che induce il robot a muoversi lungo il perimetro
    • Il metodo stop può essere visto come una interruzione della azione proattiva
    • Il metodo resume può essere considerato analogo a una return from interrupt che induce l'applicazione a proseguire dal punto in cui era stata interrotta.
    • Con riferimento al codice sviluppato nello Sprint1, il punto significativo in cui l'applicazione può subire lo stop è il durante l'esecuzione di walkAheadUntilCollision oppure di walkByStepping
    • KEY POINT-AP2 Muovere il robot per passi, come fa walkByStepping, suggerisce che lo stop (l'interrupt) possa essere gestito tra uno step e l'altro. Più difficile pensare di rendere interrompibile walkAheadUntilCollision.
    • Realizzare una console remota si presenta un problema nel problema, in quanto implica il passaggio concettuale da sistemi ad oggetti che interagiscono a procedre-call a sistemi ad 'enti attivi' che interagiscono a messaggi attraverso la rete.
    • KEY POINT-AP3 In questo sprint, al fine di realizzare un protototipo in tempi brevi , si ritiene opportuno introdurre la console come POJO locale, riservando lo sviluppo della console remota allo Sprint3.
    • Il main program dello Sprint2 deve essere visto come un configuratore del sistema.

    Architettura logica

  • La figura non mostra il main program che funge da configuratore.

    Test plans

  • KEY POINT-TP1. Appl1Core viene considerata osservabile per delegare a qualche observer il compito di effettuare controlli di correttezza a fronte delle informazioni emesse da Appl1Core.
  • KEY POINT-TP2. L'insieme di informazioni emesse da Appl1Core (linguaggio emittedInfo) per poter costruire il path del robot è il seguente: 
    robot-stepdone : il robot ha effettuato uno step con successo
robot-turnLeft : il robot ha effettuato una rotazione a sinistra
robot-collision : il robot ha incontrato un ostacolo (durante uno step)
robot-athomebegin : il robot è in HOME all’inizio dell’applicazione
robot-athomeend : il robot è in HOME al termine dell’applicazione
